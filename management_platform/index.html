Challenge 1 - Background Reading
Section 1 — IT/OT Convergence IT/OT Convergence and the Flat Network Problem
▼

What Are IT and OT Networks?

Information Technology (IT) refers to the systems used to process, store, and transmit data: servers, workstations, databases, and corporate applications. Operational Technology (OT) refers to hardware and software that monitors or controls physical equipment and processes: Programmable Logic Controllers (PLCs), sensors, Human-Machine Interfaces (HMIs), and Supervisory Control and Data Acquisition (SCADA) systems.[1]

Historically, OT networks were physically isolated from corporate IT networks, a configuration known as an air gap. This isolation was the primary security boundary. Because OT systems were never designed to be connected to external networks, they lack many of the security features present in modern IT systems: authentication, encryption, and patch management are often absent or minimal.[1]

Industry 4.0 and the End of the Air Gap

As industrial environments modernized under the concept of Industry 4.0, organizations began connecting OT systems to corporate IT networks to enable remote monitoring, data analytics, and centralized management. This eliminated the air gap and introduced IT-level network exposure to OT devices that were never designed to handle it.[2]

In a flat network, all devices across all zones share the same network segment and can communicate with each other without restriction. A corporate workstation and a PLC controlling a production line have the same level of network access. This means that if any device on the network is compromised, an attacker can move freely to any other device, including critical industrial controllers.

Why This Is a Security Problem

NIST Special Publication 800-82 describes ICS security as distinct from conventional IT security for several key reasons:[1]

  • Safety consequences: A failure in an OT system can cause physical damage, equipment destruction, or safety incidents affecting personnel.
  • Availability requirements: OT systems often must operate continuously (24/7). Downtime for patching or rebooting is frequently not acceptable.
  • Legacy systems: Many OT components run end-of-life operating systems (such as Ubuntu 18.04 LTS, which reached end of life in April 2023) and cannot be updated without replacing expensive hardware.
  • Long lifecycles: Industrial equipment has operational lifespans of 20 to 30 years, far exceeding typical IT refresh cycles.

These characteristics mean that the standard IT security approach of regular patching and frequent updates cannot be directly applied to OT environments. Network segmentation becomes the primary compensating control.

Section 2 — NIST SP 800-82 / IEC 62443 Network Segmentation: Concepts and Models
▼

What Is Network Segmentation?

Network segmentation is the practice of dividing a network into distinct zones and controlling the traffic that flows between them. Each zone groups devices that share a similar function, trust level, and security requirement. Traffic between zones is controlled by a firewall or similar enforcement point, and by default, all cross-zone traffic is denied unless explicitly permitted.[1]

The goal is to limit the blast radius of a security incident: if an attacker compromises a device in one zone, segmentation prevents them from freely reaching devices in other zones.

The Purdue Reference Model

The Purdue Reference Model defines a hierarchical architecture with five levels. It is referenced in NIST SP 800-82 as the standard framework for understanding the boundaries between enterprise and control layers.[1][3]

Level Zone Name Examples
L4Enterprise ZoneCorporate IT, ERP, email
L3Manufacturing Operations ZoneHistorians, MES, supervisory servers
L2Supervisory Control ZoneHMIs, SCADA servers, engineering workstations
L1Basic Control ZonePLCs, RTUs, DCS controllers
L0Field Devices ZoneSensors, actuators, valves

The most critical boundary is between Level 3 and Level 4: this is where IT and OT meet, and it is where a DMZ (Demilitarized Zone) should be placed to mediate all traffic.

Defense-in-Depth

NIST SP 800-82 recommends a Defense-in-Depth approach for ICS networks.[1] This means applying multiple layers of security controls so that the failure of one layer does not expose critical assets. In the context of network segmentation:

  • The first layer is the perimeter firewall between the corporate network and the DMZ.
  • The second layer is the firewall between the DMZ and the OT network.
  • Additional layers include host-based firewalls on OT devices, intrusion detection systems, and strict access controls.

Each boundary forces an attacker to defeat an additional control, and each crossing creates an opportunity for detection.

Zone and Conduit Model (IEC 62443)

The IEC 62443 standard introduces the concept of zones and conduits.[2]

  • A zone is a group of assets that share the same security requirements and trust level.
  • A conduit is the communication channel between two zones, including all the controls that govern what traffic is allowed to pass.

Every conduit must be explicitly designed, and all traffic not defined in the conduit policy must be denied by default. This is the allowlist model: block everything, then permit only what is required for operations.

Section 3 — MITRE ATT&CK for ICS Threat Landscape: How Attackers Move from IT to OT
▼

Lateral Movement and Network Pivoting

MITRE ATT&CK for ICS documents tactics and techniques used by threat actors against industrial control systems. Technique T0886 (Remote Services) describes how adversaries use legitimate remote services to move through a network after gaining initial access.[4]

In a flat network, an attacker who gains access to a corporate workstation (through a phishing email or a compromised credential) can immediately reach OT assets using standard network scanning tools. No additional exploitation is needed: the network itself provides unrestricted access.

With segmentation in place, the attacker must cross a firewall boundary. This requires either exploiting a permitted conduit, compromising a device inside the OT zone directly, or finding a misconfigured rule. Each step takes more time, generates more detectable network activity, and requires more sophisticated capabilities.

Real-World Examples

The 2015 Ukraine Power Grid attack demonstrated this attack path. Attackers compromised corporate IT systems using spear phishing, then moved laterally through the network and reached ICS components used to control substation breakers, causing power outages affecting approximately 225,000 customers. NIST SP 800-82 cites this incident as a key motivator for enforcing strict IT/OT segmentation.[1]

The Triton/TRISIS malware (2017) targeted Safety Instrumented Systems (SIS) in an oil and gas facility. The attackers had been present in the IT network for an extended period before pivoting to the OT network and reaching the SIS controllers. The absence of robust segmentation between IT and OT layers was a contributing factor.[4]

Protocol-Level Risk: Modbus

PLCs commonly use the Modbus protocol on TCP port 502. Modbus was designed in 1979 for serial communication in isolated environments and has no built-in authentication, no encryption, and no authorization mechanism.[1]

Any device on the network that can reach port 502 of a PLC can send it commands, including commands to stop a process, change setpoints, or disable safety functions. This is why blocking unauthorized network access to OT devices is critical even before any application-layer security is considered.

Section 4 — Technical Reference Firewall Configuration: iptables on Linux
▼

What Is iptables?

iptables is a Linux kernel-level packet filtering framework. It evaluates network packets against a set of rules and decides whether to accept, drop, or forward each packet. It is the standard tool for configuring firewall rules on Linux systems acting as routers or gateways.[5]

Rules are organized into tables, and within each table into chains. For traffic that passes through a Linux host acting as a router (from one network interface to another), the relevant chain is FORWARD.

Chains and Default Policies

Each chain has a default policy: the action taken when no rule matches a packet.

  • ACCEPT by default: all packets are allowed unless a rule blocks them (blocklist model). This is insecure as a default for a segmentation firewall.
  • DROP by default: all packets are silently discarded unless a rule allows them (allowlist model). This is the secure default for ICS network boundaries.

Setting the default policy to DROP establishes the allowlist model:

iptables -P FORWARD DROP

After this command, only packets that match an explicit ACCEPT rule will be forwarded. All other traffic is silently dropped.

Stateful Packet Inspection

iptables supports stateful inspection through the connection tracking module (conntrack). A stateful rule can distinguish between:

  • NEW: A packet starting a new connection.
  • ESTABLISHED: A packet belonging to a connection that was already accepted.
  • RELATED: A packet related to an existing connection (for example, an ICMP error response).

The following rule is required for TCP to work correctly after a DROP default policy, because it allows the return traffic for connections that were already established:

iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT

Without this rule, only the outbound packets of a permitted connection would be forwarded; the server's replies would be dropped, breaking TCP.

Applying Segmentation Rules (Step by Step)

1. Set the default FORWARD policy to DROP
iptables -P FORWARD DROP
2. Allow return traffic for established connections
iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
3. Block traffic between the Corporate zone and the OT zone (both directions)
iptables -A FORWARD -s 172.20.0.0/24 -d 172.22.0.0/24 -j DROP
iptables -A FORWARD -s 172.22.0.0/24 -d 172.20.0.0/24 -j DROP
4. Verify the applied rules
iptables -L FORWARD -v --line-numbers

The -v flag shows packet and byte counters (useful for confirming that rules are matching traffic). The --line-numbers flag shows the rule number, which is needed if a rule must be deleted or reordered.

Rule Order Matters

iptables evaluates rules in order from top to bottom and stops at the first matching rule. If a broad ACCEPT rule appears before a specific DROP rule, the DROP rule will never be reached. Always verify the order with iptables -L FORWARD --line-numbers after making changes.

Rollback

To remove all rules from the FORWARD chain and return to an open state, run:

iptables -F FORWARD

This flushes all rules from the chain. The default policy set with -P remains in effect, so if it was set to DROP, the chain will still drop everything until the policy is also reset.

Section 5 — NIST SP 800-30 Risk Assessment in an OT Context
▼

Why Risk Assessment Matters Before Making Changes

Before applying any change to a production network, particularly an OT network, a formal risk assessment must be conducted. This serves two purposes: it ensures that the impact of the change is understood before it is applied, and it creates an auditable record that the change was authorized based on documented reasoning.[1][6]

A risk assessment for a firewall change in an OT environment must consider both the risk of not making the change (the current exposure) and the risk of making the change (potential disruption to operational processes that depend on network connectivity).

Risk Components

A standard risk assessment considers three components:[6]

  • Threat: Who or what could exploit the vulnerability? In this lab, a threat could be an attacker who has compromised a corporate workstation and is attempting to reach the OT zone.
  • Vulnerability: What weakness exists that the threat could exploit? The absence of network segmentation means any device on the corporate network can reach the PLC and Legacy OS without restriction.
  • Impact: What would the consequence be if the vulnerability were exploited? For OT assets, impact categories include production downtime, equipment damage, safety incidents, and regulatory consequences.

Risk is a function of likelihood (how probable is it that the threat exploits the vulnerability?) and impact (how severe are the consequences?). High-exposure OT assets with no access controls and known protocol weaknesses represent a high-likelihood, high-impact scenario.

Change Management

In operational environments, firewall rule changes are classified as changes to production systems and require formal authorization before implementation. A change request typically documents:

  • A description of the change and the specific rules to be applied.
  • The business justification (the risk being mitigated).
  • The expected impact on operations, including any temporary downtime.
  • A rollback plan: the exact steps to undo the change if something goes wrong.
  • Approval from a responsible authority before the change is made.
Section 6 — Validation Confirming the Fix Works
▼

Testing Connectivity After Segmentation

After applying firewall rules, it is essential to verify that they work as intended. This involves both positive testing (blocked traffic is actually blocked) and negative testing (permitted traffic still works).

Test that the PLC is no longer reachable from the Corporate zone
ping -c 3 172.22.0.11

If segmentation is working correctly, this should produce 100% packet loss or a "Destination Host Unreachable" message.

Test that general connectivity is unaffected
ping -c 3 8.8.8.8

This confirms that the FORWARD rules are not inadvertently blocking traffic that should be allowed.

Reading iptables Output

The output of iptables -L FORWARD -v --line-numbers has the following structure:

Chain FORWARD (policy DROP) num pkts bytes target prot source destination 1 45 3780 ACCEPT all anywhere anywhere state RELATED,ESTABLISHED 2 0 0 DROP all 172.20.0.0/24 172.22.0.0/24 3 0 0 DROP all 172.22.0.0/24 172.20.0.0/24
  • policy DROP: The default action when no rule matches.
  • pkts/bytes: Counters showing how much traffic each rule has matched. A value of 0 for a DROP rule before testing means no traffic has yet attempted to cross that boundary.
  • target: The action taken (ACCEPT or DROP).
  • source/destination: The network ranges the rule applies to.
