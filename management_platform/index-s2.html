Challenge 3 -Background Reading
Section 1 -IoT API Security What Is the IoT API and Why Is It a Target?
▼

What Is This API?

The IoT network (172.21.0.0/24) contains an IoT sensor (172.21.0.11) that produces telemetry every 5 seconds, and an IoT API server (172.21.0.10) that exposes that data over HTTP on port 80. Any device that can reach port 80 gets a valid response - no credentials required.

After Challenge 1, the OT zone is isolated from corporate devices. However, the IoT zone remains reachable from the corporate network by design, to allow monitoring. The firewall explicitly permits 172.20.0.0/24 → 172.21.0.10:80. The problem is that the API has no authentication: any reachable device gets data simply by sending an HTTP request.

Why Unauthenticated APIs Are Dangerous

OWASP publishes the API Security Top 10. The second risk, API2:2023 - Broken Authentication, covers APIs that have no mechanism to verify who is making a request.[2] An API with no authentication at all is a textbook case: any request is treated as legitimate.

In an IoT context this is particularly serious because the data is often operational. Sensor readings from a production line reveal schedules, machine states, and operating parameters. In more critical deployments, an unauthenticated API might also accept write requests that change setpoints or trigger actions on physical devices.

NIST SP 800-82 Rev 3 identifies unauthorised access to OT monitoring interfaces as a significant attack vector, particularly when IoT devices are connected to systems with physical consequences.[1]

What an Attacker Can Do

  • Enumerate the API. Find all endpoints and understand what data is available.
  • Harvest operational data. Continuously poll to build a picture of normal operations for timing an attack.
  • Replay or inject requests. If the API accepts POST/PATCH, modify sensor data or trigger actions without credentials.
  • Pivot to other systems. Internal hostnames, software versions, or config details may reveal deeper attack paths.
Section 2 -Authentication Concepts Token-Based Authentication, API Keys, and JWTs
▼

Authentication vs. Authorization

Authentication verifies who you are: "Is this request coming from someone with valid credentials?" Authorization verifies what you are allowed to do: "Does this authenticated user have permission to access this resource?" This challenge focuses on authentication.

Token-Based Authentication

The principle is simple: (1) The server issues a token to a client that has authenticated. (2) The client includes the token in every subsequent request, typically in the HTTP Authorization header. (3) The server validates the token on each request and only responds if the token is valid.

TypeDescription
API KeyA static secret string. Simple to implement. Valid until manually revoked. Used when the client is a machine.
JWTA structured, signed token encoding claims (user ID, roles, expiry). Server verifies the cryptographic signature without a database lookup.
Bearer TokenAny token sent as Authorization: Bearer <token>. Defined in RFC 6750.[3]

This lab uses a static token (similar to an API key). The token is stored in /app/token.txt on the API server. If the file exists, every request must include it as a Bearer token. If not, the API runs unauthenticated.

JWT Structure

A JWT has three dot-separated Base64 parts: header.payload.signature. The server creates the signature with a secret key. On receipt, it recomputes the signature and rejects any token whose payload was tampered with. This makes JWTs self-contained and tamper-evident.[4]

NIST Guidance

NIST SP 800-63B requires that authentication secrets be generated with at least 112 bits of entropy, transmitted only over encrypted channels (HTTPS in production), and verified on every request.[5] In this lab we use HTTP for simplicity, but a production deployment would mandate HTTPS.

Section 3 -Testing APIs Using curl to Test API Authentication
▼

Basic Unauthenticated GET

curl http://172.21.0.10/sensor/data

Before any fix: returns sensor data (200 OK). After adding authentication: returns 401 Unauthorized.

Sending a Bearer Token

curl -H "Authorization: Bearer <token>" http://172.21.0.10/sensor/data

The -H flag adds a custom HTTP header. Replace <token> with the actual token string.

Checking the HTTP Status Code

Use curl -i to include response headers. The first header line shows the status code:

  • 200 OK: request succeeded, data returned.
  • 401 Unauthorized: rejected, no valid token provided.
  • 403 Forbidden: token valid but lacks permission.

To see only the status code: curl -s -o /dev/null -w "%{http_code}" http://172.21.0.10/sensor/data

Testing a Wrong Token

curl -i -H "Authorization: Bearer wrong-token" http://172.21.0.10/sensor/data

Expected: 401 Unauthorized. A correct implementation rejects wrong tokens, not just missing ones.

Section 4 -Technical Implementation How the Token Check Works in the Flask App
▼

The API Stack

The IoT API (172.21.0.10) is a Python Flask application. Authentication is controlled by the file /app/token.txt. On every request, the app checks if this file exists:

  • If the file does not exist → auth disabled, request proceeds (current state).
  • If the file exists → auth enabled; the Authorization: Bearer <token> header is checked against the file contents.

The Token Check Logic

For every incoming request: read the Authorization header → check it starts with Bearer → extract the token string → compare to the file contents → if it matches, allow; if not, return HTTP 401.

The Sensor Client

The IoT sensor (172.21.0.11) polls the API every 5 seconds using a Python script at /app/sensor.py. It also reads /app/token.txt and if the file exists, it adds the Authorization header to every request.

Critical Order of Operations

Before enabling authentication on the server, you must configure the sensor client to include the token. If you enable auth on the server first, the sensor immediately starts receiving 401 errors. Configure and restart the sensor first, verify it is working, then enable auth on the API second.

This is why the change management process exists: to ensure all dependent clients are ready before the breaking change is applied.

Section 5 -Residual Risk What Remains After the Fix
▼

Residual Risk After Adding Token Auth

  • HTTP, not HTTPS: the token is transmitted in plaintext. A network sniffer on the same segment can capture it. In production, HTTPS would be mandatory.
  • Static token, no expiry: if leaked, the token remains valid until manually revoked. JWT tokens with an exp field reduce this risk.
  • Token stored on disk - /app/token.txt on both the API server and sensor containers. If either container is compromised, the token is exposed.

Documenting residual risk is part of a proper risk acceptance decision. The approving authority must understand what risk remains after the fix and explicitly accept it.[6]

Rollback Plan How to revert API authentication if something goes wrong
▼

When would you roll back?

If enabling authentication causes an unexpected issue - for example, a monitoring tool you forgot to update starts receiving 401 errors and stops working - you can disable authentication immediately without rebuilding or restarting the containers.

Rollback commands - run on IoT API container

Remove the token file (disables auth on the next request cycle):

rm /app/token.txt

Restart the API to ensure the change takes effect immediately:

pkill -f app.py; python /app/app.py &

Verify auth is disabled:

curl -s http://localhost/health

The response should show "auth_enabled": false.

Also roll back the sensor (optional)

If you also need to revert the sensor to unauthenticated requests:

rm /app/token.txt; pkill -f sensor.py; python /app/sensor.py &

Why this matters

Because the token file controls authentication at runtime, rollback takes seconds and does not require any container rebuild. This is why the change management step asks you to document the rollback procedure before applying the fix.
