Section 1 — EOL Systems End-of-Life Operating Systems in OT Environments
▼

What Does "End of Life" Mean?

An operating system reaches End of Life (EoL) when the vendor stops releasing security patches and updates. From that point forward, any newly discovered vulnerability in the OS will never be officially fixed. Ubuntu 18.04 LTS reached end of life in April 2023.[1]

In a standard IT environment, the response to an EoL system is to upgrade it. In OT environments, this is often not possible:

  • Industrial software may only be certified to run on a specific OS version.
  • Hardware vendors may not support newer OS versions on older field devices.
  • Upgrading requires extensive testing to ensure operational continuity, which is expensive and disruptive.
  • Some systems are embedded and cannot be modified at all without replacing the hardware.

The result is that EoL systems remain in production for years or decades beyond their support window. NIST SP 800-82 Rev 3 explicitly acknowledges this constraint and recommends compensating controls as the practical alternative to patching.[1]

The Risk of an Unpatched OT System

Known vulnerabilities for an EoL OS accumulate over time and are publicly documented in the Common Vulnerabilities and Exposures (CVE) database. Attackers use this database to find reliable exploit code for systems they have identified through network scanning. An EoL system is effectively a system with a permanently growing list of known, unmitigated attack paths.

When an EoL system is also running insecure services (like telnet or SMBv1), the attack surface is compounded: not only is the OS vulnerable, but the services themselves provide additional direct entry points.

Compensating Controls

A compensating control is a security measure that reduces risk when the primary control (patching) cannot be applied. For EoL OT systems, typical compensating controls include:[1]

  • Network isolation: Restrict which devices can reach the legacy system (already applied in Challenge 1).
  • Service hardening: Disable all services that are not required for operations.
  • Host-based firewall: Apply iptables INPUT rules on the legacy host itself to limit inbound connections.
  • Monitoring: Log all access attempts and alert on anomalous connections.
Section 2 — CVE-2017-0144 / SMBv1 SMBv1 and the EternalBlue Vulnerability
▼

What Is SMBv1?

Server Message Block version 1 (SMBv1) is a network file-sharing protocol first implemented in 1983. It was designed for small, isolated networks and lacks modern security features: no message signing by default, no encryption, and several design-level weaknesses. All modern operating systems have deprecated or removed SMBv1, but legacy systems may still enable it.[2]

CVE-2017-0144 — EternalBlue

CVE-2017-0144, known as EternalBlue, is a critical remote code execution vulnerability in the SMBv1 implementation. It was originally developed by the NSA, leaked by the Shadow Brokers group in April 2017, and subsequently weaponized in two of the most damaging cyberattacks in history:[2][3]

  • WannaCry (May 2017): A ransomware worm that used EternalBlue to spread autonomously across networks, infecting over 200,000 systems in 150 countries. It caused an estimated $4–8 billion in damages and severely impacted the UK National Health Service.
  • NotPetya (June 2017): A destructive wiper disguised as ransomware that used EternalBlue for lateral movement. It caused over $10 billion in damages and is considered one of the most costly cyberattacks ever recorded.

Both attacks demonstrated that a single unpatched SMBv1 system on a network can allow an attacker (or automated worm) to compromise every other vulnerable system on the same segment — classic lateral movement with no authentication required.

Mitigation

Disabling SMBv1 eliminates exposure to EternalBlue and related exploits. In Samba on Linux, this is achieved by setting a minimum protocol version:

echo "min protocol = SMB2" >> /etc/samba/smb.conf

This forces all SMB connections to use version 2 or higher, which does not have the EternalBlue vulnerability.

Section 3 — Insecure Protocols Why Telnet Is Dangerous in an OT Environment
▼

Telnet and Cleartext Transmission

Telnet (TCP port 23) is a protocol from 1969 that provides a remote command-line interface. Every character typed in a telnet session — including usernames, passwords, and commands — is transmitted as unencrypted plaintext across the network. Any device on the same network segment that can capture traffic (via a network tap, port mirroring, or ARP spoofing) can read all telnet credentials and commands in real time.[4]

In MITRE ATT&CK for ICS, this maps to technique T1040 (Network Sniffing): adversaries capture network traffic to obtain credentials or observe control commands sent between systems.[3]

Why This Is Especially Risky in OT

In an OT network, telnet credentials captured from a maintenance connection to a legacy system can be reused to:

  • Access the system as a legitimate user, bypassing all network-level access controls.
  • Issue commands that change process parameters or stop running services.
  • Create persistence by adding new accounts or modifying startup configurations.

The secure replacement for telnet is SSH (Secure Shell), which encrypts the entire session including credentials. However, for legacy OT systems that cannot run SSH, the only compensating control is to ensure telnet is not exposed at all — by disabling the service and enforcing strict network access controls.

Disabling Telnet

Telnet on Ubuntu 18.04 is managed by openbsd-inetd, the internet super-server daemon:

service openbsd-inetd stop

After stopping the service, verify that port 23 is no longer listening:

ss -tlnp | grep :23

An empty result confirms the service is down.

Section 4 — Technical Reference Host-Based Firewalling: iptables INPUT Chain
▼

Host-Based vs. Network-Based Firewall

In Challenge 1, you applied firewall rules on the network router (the FORWARD chain). This controls traffic that passes through the router between zones. A host-based firewall is different: it controls traffic that is destined for or originating from the host itself. It is applied on the individual machine, not the router.[1]

Defense-in-depth means both are active simultaneously. Even if an attacker bypasses the network firewall (for example, by compromising another device already inside the OT zone), the host-based firewall provides a second layer of protection.

The INPUT Chain

The iptables INPUT chain processes packets destined for the local machine. To restrict who can connect to the legacy OS:

Allow traffic from within the OT zone (PLC can still communicate)
iptables -A INPUT -s 172.22.0.0/24 -j ACCEPT
Allow established return traffic
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
Always allow loopback traffic
iptables -A INPUT -i lo -j ACCEPT
Drop everything else (default deny)
iptables -P INPUT DROP

Important: Rule Order

The ACCEPT rules must be added before setting the default policy to DROP — otherwise the iptables command that adds the ACCEPT rules itself may be blocked. Always verify the final state:

iptables -L INPUT -v --line-numbers

Verifying the Effect

After applying host-based firewall rules, check that intra-OT communication still works (PLC to Legacy OS), while connections from other zones are dropped. This validates that the compensating control does not break legitimate operational traffic.

Section 5 — NIST SP 800-82 Compensating Controls Framework
▼

When Patching Is Not Possible

NIST SP 800-82 Rev 3 dedicates a section to the challenge of legacy and unsupported systems in ICS. The guidance acknowledges that patching is not always feasible and outlines a compensating control strategy:[1]

  • Isolate: Apply network segmentation to limit the blast radius if the system is compromised (done in Challenge 1).
  • Harden: Disable all services not required for operations. Reduce the attack surface to only what is necessary.
  • Monitor: Log all access to legacy systems and alert on anomalies. Compensating controls rely heavily on detection when prevention is incomplete.
  • Document: Record the unmitigated vulnerabilities, the compensating controls in place, and the residual risk. This is an auditable risk acceptance decision.

Section 6 — Validation Confirming the Hardening Worked
▼

Verify Services Are Stopped

After disabling telnet, confirm port 23 is no longer listening on the legacy OS:

ss -tlnp

Port 23 should not appear in the output. An empty result for grep :23 confirms success.

Verify Samba Configuration

grep "min protocol" /etc/samba/smb.conf

The output should show min protocol = SMB2 or higher.

Verify Host-Based Firewall

iptables -L INPUT -v --line-numbers

Confirm that ACCEPT rules for the OT zone and loopback are present, and the default policy is DROP.

Confirm Intra-OT Traffic Still Works

From the PLC container, the legacy OS should still be reachable (they are in the same OT zone). This verifies that the compensating controls do not break legitimate operational communication.
